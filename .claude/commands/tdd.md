Issue-Driven TDD ワークフロースキル

## 引数
$ARGUMENTS — issue ファイル名または番号（例: `001`, `001_scheduler-initialization.md`）

## ワークフロー

### Phase 1: Issue 読み込みと受け入れ基準の策定

1. `issues/` ディレクトリから該当 issue ファイルを読み込む
   - 番号のみ指定された場合はパターンマッチで検索（例: `001` → `issues/001_*.md`）
2. issue の内容を分析し、以下を明確化:
   - **ゴール**: この issue が完了した状態の定義
   - **受け入れ基準（Acceptance Criteria）**: 具体的・検証可能な条件をリストアップ
   - **スコープ外**: 今回やらないことの明示
3. 受け入れ基準を issue ファイルの `## 受け入れ基準` セクションに追記する
4. ユーザーに受け入れ基準のレビューを依頼し、承認を得る

### Phase 2: テスト設計（TDD の Red フェーズ）

1. 受け入れ基準の各項目に対応するテストケースを設計する
2. テストファイルを作成する（`tests/test_{issue_name}.py` または既存テストファイルに追加）
3. テストの方針:
   - 受け入れ基準 1 項目 = 1 テスト関数以上
   - テスト名は `test_ac{番号}_{内容}` 形式（例: `test_ac1_scheduler_starts_on_api_launch`）
   - 外部 API は必ずモック化
   - 正常系・異常系の両方をカバー
4. テストを実行し、**全てが FAIL することを確認**（Red）
5. ユーザーにテスト項目を提示し、確認を得る

### Phase 3: 実装（TDD の Green フェーズ）

1. テストを 1 つずつパスさせる順序で実装を進める
2. 実装ステップごとに:
   - 対象テストを明示
   - 最小限のコード変更で当該テストをパスさせる
   - テスト実行して Green を確認
   - 次のテストへ進む
3. 全テストが Green になるまで繰り返す

### Phase 4: リファクタリング（TDD の Refactor フェーズ）

1. 全テストが Green の状態で:
   - コードの重複除去
   - 命名改善
   - 不要な複雑さの除去
2. リファクタリング後もテストが全て Green であることを確認

### Phase 5: 完了処理

1. 全テスト Green を最終確認
2. issue ファイルの「やるべきこと」チェックリストを更新（完了項目にチェック）
3. issue ファイルに `## 完了` セクションを追加:
   - 完了日
   - 実装サマリ
   - 変更ファイル一覧
4. コミットメッセージは `feat: #{issue番号} {概要}` 形式
5. プッシュ

## ルール

- **テストが先。実装は後。** テストなしにプロダクションコードを書かない
- 受け入れ基準に含まれないことは実装しない（スコープクリープ防止）
- 各フェーズの完了時にユーザーに報告し、次のフェーズに進む許可を得る
- テストが Red のまま次のテストに進まない — 1 つずつ Green にする
- 外部 API 呼び出しは全てモック化し、CI 環境でも実行可能にする
